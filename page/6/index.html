2. Why is the app killed shortly after locking the screen after creating a child window?

Long-running tasks must be enabled to keep the app alive when the screen is locked or the app is sent to the background.

If an app needs to perform user-perceivable long-term tasks in the background after being sent to the background, such as playing music or navigation, it can apply for a long-running task to prevent the system from suspending the app’s process and causing functional anomalies.

After applying for a long-running task, the system performs corresponding checks to ensure the app is executing the task. There will also be a notification in the notification bar associated with the long-running task. If the user deletes this notification, the system will automatically stop the long-running task.

Open Anonymous Device Identifier (OAID): A non-permanent device identifier that enables personalized advertising while protecting user data privacy. Third-party monitoring platforms can also provide conversion attribution analysis to advertisers based on OAID.
Developers of media apps, advertising platforms, and third-party monitoring platforms can obtain the OAID on the device for personalized ad recommendation or ad conversion attribution analysis.

OAID is a 32-bit UUID-like identifier generated by Huawei’s proprietary algorithm, formatted as xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx.
OAID characteristics:

  • Device-level identifier: The same OAID is obtained by different apps on the same device.
  • Acquisition is affected by the “Cross-App Association Access Permission” switch:
    • When the switch is on, the app obtains a valid non-all-zero OAID.
    • When the switch is off, the app can only obtain an all-zero OAID.
  • The OAID is generated for the first time when the first app on the device enables the “Cross-App Association Access Permission” switch.

2. How to implement mobile card top-up via NFC in HarmonyOS?

Required capabilities: Using the NFC adapter to establish connections, send and receive data. The corresponding business scenario is using the NFC antenna to top up transit cards, involving NFC reading/writing of CPU cards based on the ISO-14443-4 protocol specification.

Standard NFC-Tag tag.TagInfo demo: 

import tag from '@ohos.nfc.tag';
import UIAbility from '@ohos.app.ability.UIAbility';
import AbilityConstant from '@ohos.app.ability.AbilityConstant';
import Want from '@ohos.app.ability.Want'

export default class EntryAbility extends UIAbility {
  onCreate(want : Want, launchParam: AbilityConstant.LaunchParam) {
    // add other code here...

    // want is initialized by nfc service, contains tag info for this found tag
    let tagInfo : tag.TagInfo | null = null;
    try {
      tagInfo = tag.getTagInfo(want);
    } catch (error) {
      console.error("tag.getTagInfo catched error: " + error);
    }
    if (tagInfo == null || tagInfo == undefined) {
      console.log("no TagInfo to be created, ignore it.");
      return;
    }

    // get the supported technologies for this found tag.
    let isNfcATag = false;
    let isIsoDepTag = false;
    for (let i = 0; i < tagInfo.technology.length; i++) {
      if (tagInfo.technology[i] == tag.NFC_A) {
        isNfcATag = true;
      }

      if (tagInfo.technology[i] == tag.ISO_DEP) {
        isIsoDepTag = true;
      }
      // also check for technology: tag.NFC_B/NFC_F/NFC_V/NDEF/MIFARE_CLASSIC/MIFARE_ULTRALIGHT/NDEF_FORMATABLE
    }

    // use NfcA APIs to access the found tag.
    if (isNfcATag) {
      let nfcA : tag.NfcATag | null = null;
      try {
        nfcA = tag.getNfcATag(tagInfo);
      } catch (error) {
        console.error("tag.getNfcATag catched error: " + error);
      }
      // other code to read or write this found tag.
    }

    // use getIsoDep APIs to access the found tag.
    if (isIsoDepTag) {
      let isoDep : tag.IsoDepTag | null = null;
      try {
        isoDep = tag.getIsoDep(tagInfo);
      } catch (error) {
        console.error("tag.getIsoDep catched error: " + error);
      }
      // other code to read or write this found tag.
    }
    // use the same code to handle for "NfcA/NfcB/NfcF/NfcV/Ndef/MifareClassic/MifareUL/NdefFormatable".
  }
}

5. Inquiry about ArkTS and SO hot update in HarmonyOS

In Android, many apps require hot updates, typically via two schemes:

  1. Differential updates for Java code, SO code, and resource files.
  2. Runtime loading of external SO files via hook mechanisms.

Does HarmonyOS have similar mechanisms, especially for loading SO files from outside the package? This is intended to fix bugs in released packages that require modifying ArkTS or C/C++ code without replacing the entire package.

3. What is the order of CallState status codes when a user makes a call in HarmonyOS?

Status code changes for three scenarios when a user dials (outgoing call), including rejection, answer, or no answer:

  • Active call rejected by the other party / Caller hangs up during ringing: 2 -> 0
  • Other party answers: 2 -> 2 (status code remains 2 during ringing)

4. How to perform application data migration in HarmonyOS?

When upgrading from HarmonyOS 3.0 to the Next system, can the persistent data of an app be synchronized to an app in Hap format? If not, how should application data be migrated?

HarmonyOS NEXT provides a “Data Migration Framework” to uniformly schedule application data migration tasks.
Reference: https://developer.huawei.com/consumer/cn/doc/harmonyos-guides-V5/app-data-migration-overview-V5

When upgrading a terminal device from HarmonyOS 3.1 Release API 9 or earlier (referred to as “HarmonyOS”) to HarmonyOS NEXT Developer Preview1 or later (referred to as “HarmonyOS NEXT”), APK applications originally running on HarmonyOS need to be converted to native HarmonyOS applications (referred to as “HarmonyOS applications”). Some data from the APK applications must be converted and migrated to specified locations before they can be accessed by HarmonyOS applications. HarmonyOS NEXT provides a “Data Migration Framework” and a “Backup and Recovery Framework” to assist developers in migrating and converting application data. After developers complete the adaptation and convert APK applications to HarmonyOS applications, the data from the original APK applications that is compatible with HarmonyOS applications can be inherited.

As shown in the figure below, application data includes data stored on cloud servers, data in the local application sandbox, and data in the local public media library. To ensure data inheritance, developers must ensure that cloud data definitions are compatible with both APK and HarmonyOS applications, ensuring that data under the same account can be recognized after the system upgrade.

System Capability (SysCap) refers to each relatively independent feature in the operating system. Different devices correspond to different system capability sets, and each system capability maps to one or more APIs. Developers can judge whether an interface can be used based on system capabilities.

canIUse(syscap: string): boolean
Parameters:

Parameter Name Type Required Description
syscap string Yes Name of the system capability to query.

Return Value:

Type Description
boolean Query result of system capability. true indicates support; false indicates no support.

2. Does HarmonyOS currently support fingerprint authentication?

Is there a corresponding capability to add fingerprint login based on account-password login?

Refer to the development guide for switching to custom authentication in User Authentication Kit: https://developer.huawei.com/consumer/cn/doc/harmonyos-guides-V5/apply-custom-authentication-V5

The unified user authentication framework provides a mechanism to switch to business-customized authentication. If device manufacturers have customized other authentication capabilities and connected them to the unified user authentication framework, application developers can switch to custom authentication methods.

For example, in payment scenarios, if a manufacturer has customized a payment password authentication method, users can choose to switch to payment password authentication after system face or fingerprint authentication fails.

Payment password authentication does not belong to system authentication capabilities. Therefore, when initiating authentication, the business needs to pass explicit information about navigation keys, such as “Use Payment Password”, so that the authentication interface shown to users includes a “Use Payment Password” button.

When the user clicks this button, the business application that initiated the authentication will receive a special authentication result from the unified user authentication framework, indicating that system authentication has ended, and the business needs to launch its custom authentication interface. Thus, after clicking “Use Payment Password”, the user will see the system authentication control disappear and the business-customized payment password authentication interface displayed.

The Kit supports biometric authentication + business-customized authentication methods, with fingerprint + account-password authentication being one of them. Applications can also integrate embedded fingerprint authentication controls on the account authentication interface. See: https://developer.huawei.com/consumer/cn/doc/harmonyos-guides-V5/user-auth-icon-V5
Mainly used to provide face and fingerprint authentication icons displayed in the application interface, with specific functions as follows:

  1. Provides embedded face and fingerprint authentication control icons for application integration.
  2. Supports customizing icon color and size, but the icon style cannot be changed.
  3. Clicking the control icon can launch the system pop-up face or fingerprint authentication control.

Currently supported authentication types for using authentication controls include:

  • Face authentication
  • Fingerprint authentication
  • Face + lock screen password authentication
  • Fingerprint + lock screen password authentication
  • Face + fingerprint + lock screen password authentication

loadRawFd(context: Context, rawFileName: string) {
  context.resourceManager.getRawFd(rawFileName, (err: BusinessError, value: resourceManager.RawFileDescriptor) => {
    let fd = value.fd;
    let offset = value.offset;
    let fdLength = value.length;
    log.info('fd: ' + value.fd + ', offset: ' + offset + ', length: ' + fdLength);
    let stream: fs.Stream = fs.fdopenStreamSync(fd, 'r');

    let stringBuffer = new ArrayBuffer(128); // Default capacity of 128 bytes per read
    let currentLength = 0; // Current read length
    let readLength = 128; // Default read length of 128 bytes
    while (currentLength < fdLength) {
      readLength = (fdLength - currentLength > 128) ? 128 : (fdLength - currentLength);
      stringBuffer = new ArrayBuffer(readLength);
      stream.readSync(stringBuffer, { offset: offset + currentLength, length: readLength });
      let decoder = util.TextDecoder.create('utf-8');
      let str = decoder.decodeWithStream(new Uint8Array(stringBuffer));
      log.info(str); // Parse to string and print log
      currentLength += readLength; // Update read start position
    }

    context.resourceManager.closeRawFdSync(rawFileName); // Close FD after reading
    // fs.closeSync(fd); // Do not use this API; optimization guidance has been provided to R&D.
    stream.closeSync();
  });
}

1. How does HarmonyOS solve the issue where voice chat and communication apps stop system collection and playback callbacks when moved to the background, and fail to resume when returning to the foreground?

The reason why app callbacks for system collection and playback stop when the app is moved to the background is as follows: To reduce device power consumption and ensure smooth user experience, the system manages background apps by suspending processes (i.e., the system no longer allocates CPU resources to the app process, and corresponding public events are no longer sent to the app process) or terminating processes.

5. Questions about HarmonyOS application keep-alive:

  1. Does an app need to apply for only one background task? For example, when an app is in the background on certain pages, does it need one background task for keep-alive, or does each page need a separate application?
  2. What is the API for background keep-alive?

Answers:

  1. You can apply for one or multiple types of long-term tasks at the same time. For multi-type services, only the main business type needs to be applied for. As long as the main task exists, the app will not be frozen.
  2. For background long-term tasks, refer to the document: https://developer.huawei.com/consumer/cn/doc/harmonyos-guides-V5/continuous-task-V5

Constraints and Limitations for Long-Term Task Application:

  • Application Restrictions: In the Stage model, long-term tasks are only supported by UIAbility; in the FA model, they are only supported by ServiceAbility.
  • Quantity Restrictions: A UIAbility (ServiceAbility in the FA model) can only apply for one long-term task at a time, meaning a new task can only be applied for after the previous one ends. If an app needs to apply for multiple long-term tasks, multiple UIAbilities must be created. Once a UIAbility in an app applies for a long-term task, all processes under the app will not be suspended.
  • Operation Restrictions: On mobile devices, the system performs long-term task validation:
    • Scenario 1: If an app applies for a long-term task but does not actually execute the task of the applied type or the task has ended, the system will manage the app. For example, if the system detects that an app has applied for AUDIO_PLAYBACK (audio/video playback) but is not playing music, the long-term task will be canceled.
    • Scenario 2: If an app executes a long-term task of a type it has not applied for, the system will manage the app. For example, if the system detects that an app has only applied for AUDIO_PLAYBACK but is also recording (corresponding to AUDIO_RECORDING), the app will be managed.
    • Scenario 3: If the background load of the process running the long-term task continuously exceeds the typical load of the applied type, the system will manage the app.

Steps to use Charles for packet capture in the HarmonyOS operating system:

Configure Phone Proxy

  1. Select the connected Wi-Fi network, right-click, and choose Modify Network.
  2. Enable Show Advanced Options, enter the server hostname (your computer’s IP) and port, then save.

Install SSL Certificate

  1. On your computer, go to Help → SSL Proxying → Install Charles Root Certificate.
  2. If your system requires authentication, enter your credentials and follow the prompts to install the certificate.

Configure SSL Capture Domains

  1. In Charles, go to Proxy → SSL Proxying Settings… and enable Enable SSL Proxying.
  2. Set the Host field to "*" (use a specific domain if needed) and port 443.

Start Capturing

After completing these steps, Charles will capture network traffic from your phone. Ensure both devices are on the same network and developer options are enabled on the phone to install and trust the Charles certificate.
