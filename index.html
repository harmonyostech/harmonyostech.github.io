2. Why is the app killed shortly after locking the screen after creating a child window?

Long-running tasks must be enabled to keep the app alive when the screen is locked or the app is sent to the background.

If an app needs to perform user-perceivable long-term tasks in the background after being sent to the background, such as playing music or navigation, it can apply for a long-running task to prevent the system from suspending the app’s process and causing functional anomalies.

After applying for a long-running task, the system performs corresponding checks to ensure the app is executing the task. There will also be a notification in the notification bar associated with the long-running task. If the user deletes this notification, the system will automatically stop the long-running task.

Open Anonymous Device Identifier (OAID): A non-permanent device identifier that enables personalized advertising while protecting user data privacy. Third-party monitoring platforms can also provide conversion attribution analysis to advertisers based on OAID.
Developers of media apps, advertising platforms, and third-party monitoring platforms can obtain the OAID on the device for personalized ad recommendation or ad conversion attribution analysis.

OAID is a 32-bit UUID-like identifier generated by Huawei’s proprietary algorithm, formatted as xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx.
OAID characteristics:

  • Device-level identifier: The same OAID is obtained by different apps on the same device.
  • Acquisition is affected by the “Cross-App Association Access Permission” switch:
    • When the switch is on, the app obtains a valid non-all-zero OAID.
    • When the switch is off, the app can only obtain an all-zero OAID.
  • The OAID is generated for the first time when the first app on the device enables the “Cross-App Association Access Permission” switch.

export default class EntryAbility extends UIAbility {
  onCreate(want : Want, launchParam: AbilityConstant.LaunchParam) {
    // add other code here...

    // want is initialized by nfc service, contains tag info for this found tag
    let tagInfo : tag.TagInfo | null = null;
    try {
      tagInfo = tag.getTagInfo(want);
    } catch (error) {
      console.error("tag.getTagInfo catched error: " + error);
    }
    if (tagInfo == null || tagInfo == undefined) {
      console.log("no TagInfo to be created, ignore it.");
      return;
    }

    // get the supported technologies for this found tag.
    let isNfcATag = false;
    let isIsoDepTag = false;
    for (let i = 0; i < tagInfo.technology.length; i++) {
      if (tagInfo.technology[i] == tag.NFC_A) {
        isNfcATag = true;
      }

      if (tagInfo.technology[i] == tag.ISO_DEP) {
        isIsoDepTag = true;
      }
      // also check for technology: tag.NFC_B/NFC_F/NFC_V/NDEF/MIFARE_CLASSIC/MIFARE_ULTRALIGHT/NDEF_FORMATABLE
    }

    // use NfcA APIs to access the found tag.
    if (isNfcATag) {
      let nfcA : tag.NfcATag | null = null;
      try {
        nfcA = tag.getNfcATag(tagInfo);
      } catch (error) {
        console.error("tag.getNfcATag catched error: " + error);
      }

    }

        isoDep = tag.getIsoDep(tagInfo);
      } catch (error) {

      }

    }

5. Inquiry about ArkTS and SO hot update in HarmonyOS

In Android, many apps require hot updates, typically via two schemes:

  1. Differential updates for Java code, SO code, and resource files.
  2. Runtime loading of external SO files via hook mechanisms.

Does HarmonyOS have similar mechanisms, especially for loading SO files from outside the package? This is intended to fix bugs in released packages that require modifying ArkTS or C/C++ code without replacing the entire package.

Response: Based on HarmonyOS’ unified app distribution ecosystem rules, quick hotfixes via patches are not provided. Such requirements are addressed through efficient review and app version update mechanisms.
