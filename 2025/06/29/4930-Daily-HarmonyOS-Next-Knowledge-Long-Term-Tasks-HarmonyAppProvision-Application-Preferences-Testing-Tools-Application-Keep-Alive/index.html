[Daily HarmonyOS Next Knowledge] Long-Term Tasks, HarmonyAppProvision Application, Preferences, Testing Tools, Application Keep-Alive

Posted on 2025-06-29 In Daily HarmonyOS Next Knowledge

[Daily HarmonyOS Next Knowledge] Long-Term Tasks, HarmonyAppProvision Application, Preferences, Testing Tools, Application Keep-Alive

1. How does HarmonyOS solve the issue where voice chat and communication apps stop system collection and playback callbacks when moved to the background, and fail to resume when returning to the foreground?

The reason why app callbacks for system collection and playback stop when the app is moved to the background is as follows: To reduce device power consumption and ensure smooth user experience, the system manages background apps by suspending processes (i.e., the system no longer allocates CPU resources to the app process, and corresponding public events are no longer sent to the app process) or terminating processes.

It is recommended to use long-term tasks in background task development services. Apply for a long-term task in the onPageHide lifecycle of the page and cancel it in onPageShow. You can use the backgroundTaskManager.startBackgroundRunning API to apply for a long-term task.

  • Refer to the demo in the guide, preferably the Stage model demo, which includes development instructions and precautions. Link: https://developer.huawei.com/consumer/cn/doc/harmonyos-guides-V5/continuous-task-V5
  • API documentation for task application. Link: https://developer.huawei.com/consumer/cn/doc/harmonyos-references-V5/js-apis-resourceschedule-backgroundtaskmanager-V5

Stage Model Example Code:

  1. Permission Application: Apply for the ohos.permission.KEEP_BACKGROUND_RUNNING permission. For configuration, refer to declaring permissions.
  2. Declare Background Mode Type: In the module.json5 configuration file, declare the corresponding long-term task type for the UIAbility that needs to use long-term tasks (fill in the configuration item for the long-term task type in the configuration file).
"module": {
    "abilities": [
        {
            "backgroundModes": [
                // Configuration item for the long-term task type
                "audioRecording"
            ], 
        }
    ],
    ...
}
  1. Import Modules: The modules related to long-term tasks are @ohos.resourceschedule.backgroundTaskManager and @ohos.app.ability.wantAgent, with other modules imported as needed.
import backgroundTaskManager from '@ohos.resourceschedule.backgroundTaskManager';
import UIAbility from '@ohos.app.ability.UIAbility';
import window from '@ohos.window';
import AbilityConstant from '@ohos.app.ability.AbilityConstant';
import Want from '@ohos.app.ability.Want';
import rpc from '@ohos.rpc';
import { BusinessError } from '@ohos.base';
import wantAgent, { WantAgent } from '@ohos.app.ability.wantAgent';
  1. Apply and Cancel Long-Term Tasks:
    • In the Stage model, long-term tasks support applications from the same device and app. Cross-device or cross-app applications are only open to system apps. Example code for applying for a long-term task from the same device and app is as follows:
@Entry
@Component
struct Index {
  @State message: string = 'ContinuousTask';
  // Get the UIAbility context of the page through the getContext method.
  private context: Context = getContext(this);

  startContinuousTask() {
    let wantAgentInfo: wantAgent.WantAgentInfo = {
      // List of actions to be executed after clicking the notification
      // Add the bundleName and abilityName of the app to be launched
      wants: [
        {
          bundleName: "com.example.myapplication",
          abilityName: "com.example.myapplication.MainAbility"
        }
      ],
      // Specify that the action after clicking the notification is to launch the ability
      actionType: wantAgent.OperationType.START_ABILITY,
      // A private value defined by the user
      requestCode: 0,
      // Action execution attributes after clicking the notification
      wantAgentFlags: [wantAgent.WantAgentFlags.UPDATE_PRESENT_FLAG]
    };

    // Get the WantAgent object through the getWantAgent method in the wantAgent module
    wantAgent.getWantAgent(wantAgentInfo).then((wantAgentObj: WantAgent) => {
        backgroundTaskManager.startBackgroundRunning(this.context,
          backgroundTaskManager.BackgroundMode.AUDIO_RECORDING, wantAgentObj).then(() => {
          console.info(`Succeeded in operationing startBackgroundRunning.`);
        }).catch((err: BusinessError) => {
          console.error(`Failed to operation startBackgroundRunning. Code is ${err.code}, message is ${err.message}`);
        });
     });
   }

   stopContinuousTask() {
      backgroundTaskManager.stopBackgroundRunning(this.context).then(() => {
        console.info(`Succeeded in operationing stopBackgroundRunning.`);
      }).catch((err: BusinessError) => {
        console.error(`Failed to operation stopBackgroundRunning. Code is ${err.code}, message is ${err.message}`);
      });
   }

   build() {
     Row() {
       Column() {
         Text("Index")
           .fontSize(50)
           .fontWeight(FontWeight.Bold)

        Button() {
           Text('Apply for Long-Term Task').fontSize(25).fontWeight(FontWeight.Bold)
         }
         .type(ButtonType.Capsule)
         .margin({ top: 10 })
         .backgroundColor('#0D9FFB')
         .width(250)
         .height(40)
         .onClick(() => {
           // Apply for a long-term task via the button
           this.startContinuousTask();

           // Execute specific long-term task logic here, such as audio playback.
         })

         Button() {
           Text('Cancel Long-Term Task').fontSize(25).fontWeight(FontWeight.Bold)
         }
         .type(ButtonType.Capsule)
         .margin({ top: 10 })
         .backgroundColor('#0D9FFB')
         .width(250)
         .height(40)
         .onClick(() => {
           // End the execution of the specific long-term task here

           // Cancel the long-term task via the button
           this.stopContinuousTask();
         })
       }
       .width('100%')
     }
     .height('100%')
   }
 }

3. What storage solution is recommended when the data stored in HarmonyOS preferences exceeds the maximum length?

Use a key-value database—single-version database—where the key length ≤ 1 KB and the value length < 4 MB per record.
Reference document: https://developer.huawei.com/consumer/cn/doc/harmonyos-guides-V5/data-persistence-by-kv-store-V5

5. Questions about HarmonyOS application keep-alive:

  1. Does an app need to apply for only one background task? For example, when an app is in the background on certain pages, does it need one background task for keep-alive, or does each page need a separate application?
  2. What is the API for background keep-alive?

Answers:

  1. You can apply for one or multiple types of long-term tasks at the same time. For multi-type services, only the main business type needs to be applied for. As long as the main task exists, the app will not be frozen.
  2. For background long-term tasks, refer to the document: https://developer.huawei.com/consumer/cn/doc/harmonyos-guides-V5/continuous-task-V5

Constraints and Limitations for Long-Term Task Application:

  • Application Restrictions: In the Stage model, long-term tasks are only supported by UIAbility; in the FA model, they are only supported by ServiceAbility.
  • Quantity Restrictions: A UIAbility (ServiceAbility in the FA model) can only apply for one long-term task at a time, meaning a new task can only be applied for after the previous one ends. If an app needs to apply for multiple long-term tasks, multiple UIAbilities must be created. Once a UIAbility in an app applies for a long-term task, all processes under the app will not be suspended.
  • Operation Restrictions: On mobile devices, the system performs long-term task validation:
    • Scenario 1: If an app applies for a long-term task but does not actually execute the task of the applied type or the task has ended, the system will manage the app. For example, if the system detects that an app has applied for AUDIO_PLAYBACK (audio/video playback) but is not playing music, the long-term task will be canceled.
    • Scenario 2: If an app executes a long-term task of a type it has not applied for, the system will manage the app. For example, if the system detects that an app has only applied for AUDIO_PLAYBACK but is also recording (corresponding to AUDIO_RECORDING), the app will be managed.
    • Scenario 3: If the background load of the process running the long-term task continuously exceeds the typical load of the applied type, the system will manage the app.
